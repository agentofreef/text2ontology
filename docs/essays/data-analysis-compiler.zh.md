# 数据分析的编译器

> 代码能让 AI 写对,靠的是测试当裁判;数据分析没有裁判——所以三年都困在 demo。但"没有裁判"这个结论,只对了一半。

---

你让 AI 帮你查一个业务数字,它干脆利落地回你"12.3%"。停一秒 —— **你怎么知道这个 12.3% 是对的?**

你验不了。因为"早单率"里的"早单",到底是 `status='CONFIRMED'`,还是 `IN ('CONFIRMED','SHIPPED')`,这个定义只活在某个人的脑子里,不在代码、也不在数据里。换成写代码就完全不同:AI 写完,你跑一遍测试,过或不过,清清楚楚 —— **测试就是那个"告诉你对不对"的裁判**。数据分析没有这个裁判。

这就是为什么过去三年,"用大白话问你的数据库"的 demo 满天飞,真正进了企业生产的却寥寥无几:没有裁判,模型就无法收敛,永远停在"看起来很惊艳"那一步。这不是模型不够强,是这件事**结构上**就缺了一样东西。(这个"没有裁判 → 无法收敛"的完整论证,我在 [本体宣言](../manifesto/manifesto.zh.md) 里展开过——但这篇不需要你读过它。)

而"数据分析没有裁判"这个判断,其实只对了一半。**真正没法验证的,从来不是整个问题,而只是其中一刀**:那个数到底怎么算。把这一刀管死的东西,有一个精确的名字 —— **口径**(就是枪管的那个口径,一个固定的孔径,一个"这个数到底算什么"的授权定义)。

口径,就是数据分析的编译器。这篇文章讲清楚这句话 —— 以及它为什么意味着:你终于能在答错的当下,就知道它错了。

## 把欠定收敛到唯一真正欠定的对象

把那"一刀"拆开看。"早单率是多少"这个问题里,到底哪一部分才是真正欠定的?

- "按地区拆" / "只看 Q4" / "排除测试数据"——这些是**维度和过滤**,提问时说清楚就定了,不欠定。
- "要不要 JOIN 产品表"——这是**物理路径**,本体的关系图里写死了,不欠定。
- **"早单 = `status='CONFIRMED'` 还是 `IN ('CONFIRMED','SHIPPED')`"——这才是欠定的那一刀。** 它是个**度量口径**问题,答案不在数据里,在组织共识里。

过去三年所有"AI 直接对话数据"的崩溃,根因都在这最后一刀:**LLM 被迫去猜一个只存在于人脑里的口径,然后自信地把猜测当事实报出来。**

口径系统做的事,就是把这一刀**从"运行时由 LLM 猜"变成"事先由人写死、入库、可审计"**。一条口径就是某个对象上、一条最精简的度量:

```sql
select "ORDER_TYPE", sum("ORDER_QUANTITY") as "TOTAL"
from "EARLY_ORDER"
group by "ORDER_TYPE"
```

保存时它被拆成三件:**主对象**(`EARLY_ORDER`)、**度量**(`sum(ORDER_QUANTITY)`)、**基础维度**(`ORDER_TYPE`)。它只回答"算什么",不回答"怎么筛、按什么拆"——那些留给运行时。口径**描述它自己,不描述任何一次具体查询**;它就是"早单量"这个数在你们组织里的**法定定义**。

## 为什么这是编译器,而不只是语义层

dbt、Cube、LookML 早就在做"组织维护一组指标定义"。如果口径只是又一个语义层,它不值得单独写一篇文章。

口径和语义层的差别,在于多了一道**闸门**:

> **任何一个聚合的度量列,必须被某条已授权口径背书。否则系统拒绝执行,报 `NO_AUTHORIZED_METRIC`——而不是拼一个没人背书的数糊弄过去。**

这一道闸,就是"语义层"和"编译器"的分水岭。

编译器的本质不是"能生成代码",而是**"良构的程序编译,病构的程序被报错拒绝,而不是被悄悄错编译"**。口径系统恰好是这个结构:

- 问题命中某条已授权口径 ⇒ SmartQuery 引擎沿本体关系**确定性地**编译出 Postgres SQL、执行、返回——**LLM 全程没碰过一行 SQL,没见过一个 JOIN**;
- 问题命中不了任何口径 ⇒ 报"这个口径不存在 / 缺口",**诚实拒答**。

于是那个困扰了整个行业的失败模式——**"自信地答错"**——被结构性地消灭了。只剩两种结局:

> **要么答出口径定义的那个值,要么告诉你这个口径不存在。**

"自信地报一个没人背书的数"这条路,在编译器里根本不存在。这就是为什么我敢说:**口径不是更好的语义层,口径是数据分析第一次拥有的编译器。**

## 北极星不是准确,是可审计

这是全文最重要、也最容易被夸大的一节,我必须把它讲死——因为它推翻了整个赛道的默认假设。

几乎所有"AI 对话数据"产品都在优化同一个指标:**准确性**——90%、95%、更大的模型、更好的 prompt。但在一个没有裁判的领域,**"准确性"根本不是一个能被直接优化的目标**:你连"这一次对不对"都无法判定,又拿什么梯度去下降?这正是开头那条"没有裁判 → 无法收敛"的另一种说法。

所以这套系统的北极星,从第一天起就不是准确,而是**可审计**——一个更朴素、也更要命的问题:

> **我怎么知道这个答案是错的?**

这里有一个反直觉、但最关键的事实:

**一个你无法判断对错的"准确"答案,比一个你能指着说"这里错了"的、可能错的答案,更危险。**

LLM 报一个"12.3%",哪怕它 95% 的时候是对的,你也**无法在当下知道这一次是不是那错的 5%**——拿着它砍 20% 预算的 CFO,赌的是运气。而口径背书的数不一样,它**指向一条你能打开、能读、能反驳的口径**:你点开 `lakehouse_metric`,看见"早单 = `status='CONFIRMED'`",当场就能说"不对,我们还要算 SHIPPED"——然后改那一行,所有未来查询同步修正。

借波普尔一句:**一个不可证伪的断言,对决策毫无价值。** LLM 的"12.3%"不可证伪——你根本不知道拿什么去验它;口径背书的数可证伪——它锚在一条显式定义上,你能逐条质疑。**编译器给数据分析的礼物,从来不是正确性,是可证伪性。**

到这里,"一致性优先于正确性"这句口号才终于有了它的因果链:**正确性无法直接优化(没有裁判);但一致性可以——而一致性正是可审计的前提。** 同一个问题永远得到同一个答案,你才能把那个答案钉死、追溯到那条口径、再反驳它;每次都飘的答案,根本无从审计。

所以精确地说:

> **口径系统编译出来的每个数,一定相对于组织共识正确;但不一定是相对客观世界正确的。** 它不保证你的口径在业务上是对的——它保证的是:**当口径错了,你一定能知道、能定位、能一次修好。**

C 编译器没让程序员逃开"我的逻辑可能是错的",它只是让"错"变得可定位、可修复、可重现。口径对数据分析做的是同一件事:**它不让你逃开"口径可能写错了",它让"写错了"变得当场可知。** 这才是它真正在卖的东西。

## 一条口径,回答很多问法

编译器不能是僵的。口径的灵活,来自运行时对每个属性的**三态**:

- **不传** → 该列完全不出现;
- **传了、值留空** → 仅作维度展示(`group by`,不过滤);
- **传了、且有值** → 过滤 + 展示("过滤即展示")。

于是同一条"早单量"口径,一个字不改,就能回答"各订单类型的早单量""各品牌的早单量"(沿本体关系自动 JOIN 产品对象)、"Legion 品牌的早单量"——LLM 始终只是**挑口径、填参数**,从不写 SQL。

## 从编译器到构建系统:口径依赖图

有了编译器,下一步是自然的:**让口径之间相互依赖。**

把每条口径看成一个**节点**;一条边表示"口径 B 要先拿到口径 A 的产出才能算"——**而边本身承载的就是传参**:A 的某个输出列,绑定到 B 的某个参数。

一旦有了这张图,一个根本性的转变发生了:**面对一个问题,我们不再问"它有几个意图、该拆成几个子问题"**——那是过去最脆弱、最靠 LLM 猜的一步。我们只问:这个问题命中哪条**目标口径**?它的参数,哪些由问题直接给、哪些要由别的口径的产出来喂?

于是"任务拆解"不再是 LLM 的一次推理,而是**对一张策展好的口径图做确定性求值**:从目标口径反向链,逐个把未满足的参数接到能产出对应类型的上游口径,直到所有叶子参数都能由问题本身提供。这不该叫"任务拆解器",更准的名字是**口径执行器(Caliber Executor)**——拆解是图的固有属性,不是运行时的猜测。(它和 **dbt 的 `ref()` 依赖图**、**类型化函数组合**、**Datalog 反向链**是同一个形状。`lakehouse_metric.level='plan'` 这个字段,就是为它预留的种子。)

但有一条线必须守住,否则它会塌回成一个开放搜索、不受界限的"会规划的 AI"——这套范式存在的全部意义,就在于避开它:

> **依赖边必须是人策展的、带类型的;LLM 只负责选目标口径 + 填叶子参数;执行器是确定性求值器,不是开放搜索的 planner。**

守住这条线,"要么答对、要么说不存在"就从单条口径推广到了整张图:**问题可答 ⟺ 目标口径的依赖闭包能一路回填到叶子。** 闭包凑不齐,就是缺口,诚实声明。

## 所以 BI 是伪命题,substrate 在哪里

退一步看大图。

BI 当年涌出一堆工具(Cognos、Business Objects、早期 Tableau),今天回头看,真正持续吃到红利的不是那些**表面的 BI 工具**,而是**底下的 substrate**——数据仓库和转换层(Snowflake、Databricks、dbt)。BI 工具是伪命题;真正的驱动力是"数据量太大,必然要有一层基础设施来承载它"。

AI 时代正在重演这一幕:

- **表面**:"AI agentic data analyst" / "用自然语言查你的数据库"——这一层会被大模型公司自己 ship 的功能 commoditize;
- **substrate**:**组织策展好的口径 + 把人话编译成 SQL 的编译器**——这一层一旦在某个组织里立起来,上面所有的 AI 助手、报表、对话、API 都得从这里走。

所以这套系统真正占的位置,不是"又一个 AI BI 工具",而是**"NL 驱动数据分析的 substrate"**。"AI 帮你分析"是表面;"组织把口径写死,AI 只去编译"才是底座。

## 哪里会坏(它还没做完,也天生有边界)

老规矩,把刀口朝向自己:

1. **编译器检查一致性,不检查真相。** 授权了的错口径,照样被忠实编译。这套系统能保证"答案和口径一致、错误有地址",**不能保证你的口径在业务上是对的**——那永远是人的责任。
2. **新的失败模式是"假不存在"。** 口径其实有,但分词 / 召回没命中,于是误拒。错从"自信答错"挪到了"该答却拒答"——这是**严格更好**的失败(诚实 > 错答),但它把担子全压到了分词和召回的质量上。**分词依然是 60% 以上成败的命门。**
3. **派生度量和集合值是确定性最薄的地方。** 比值(早单率 = 早单 / 总量)、要不要对上游产出的集合做迭代(map)——口径依赖图在这块会从"简单 DAG"升级成"带 map/reduce 的小型数据流语言",这是最难、最值得继续打磨的部分。
4. **冷启动时编译器是空的。** 口径集为空 → 一切都拒答。这台编译器的"文法"是业务本体工程师一条条写出来的。**革命性在模型,兑现靠策展。**

## 结语

开头那半句是:数据分析没有裁判,所以 LLM 不能收敛。

这篇文章想补的是后半句:**裁判一直都在,它的名字叫口径。** 只要你愿意把每一个度量,都锚到一条人写的、可审计的口径上,数据分析就有了它的编译器——口径就是它的类型系统,口径依赖图就是它的构建系统。

代价是诚实的:它给你的从来不是"更准",是"可审计"——它编译的是"与组织共识的一致性",不一定是"客观真相"。但这恰恰是它最该被理解的地方:**在一个没有真理裁判的领域,能知道自己什么时候错、错在哪、怎么一次修好,比偶尔蒙对值钱得多。**

写代码的人花了几十年,才让"编译器"这三个字成为理所当然。数据分析的编译器,才刚刚开始。
