# 数据分析的编译器:本体与口径

> 为什么"用大白话问你的数据库"这件事,三年了还都是 demo——以及它真正应该被怎么解决。

---

## 一、客观问题:三年了,为什么还都是 demo

你让 AI 帮你查一个业务数字,它干脆利落地回你"12.3%"。

停一秒——**你怎么知道这个 12.3% 是对的?**

你验不了。"早单率"里的"早单",在你公司到底是 `status='CONFIRMED'`,还是 `IN ('CONFIRMED','SHIPPED')`?这个定义只活在某个人脑子里,不在代码,也不在数据。换成是写代码,情况完全不同:AI 写完,你跑一遍测试,过或不过,清清楚楚——**测试就是那个"告诉你对不对"的裁判**。数据分析没有这个裁判。

软件工程里有个经典概念叫 **真理裁判(test oracle)**:一个能告诉你"正确答案是什么"的黑盒。AI 编程能闭环收敛——模型出代码、跑测试、过或不过、迭代——靠的就是这件事。**数据分析没有这样的裁判**。没有裁判 = 没有迭代收敛 = 永远是 demo。

这就是为什么过去三年,"chat with your database" 的 demo 满天飞,真正进了企业生产的寥寥无几。Snowflake Cortex、Databricks Genie、各家 Text2SQL 创业公司——开源 leaderboard 上写着"准确率 90%+",但 10 个回答里有 1 个错的,当 CFO 拿着错的那一个砍 20% 预算时,谁负责?**90% 在娱乐场景下是惊艳,在决策场景下是不可用。**

这不是模型不够强,也不是 prompt 没调好。这是**范式的问题**:把"生成 SQL"这件事交给一个无法被验证的黑盒。

但"数据分析没有裁判"这个判断,只对了一半。**真正没法验证的,从来不是整个问题,而只是其中一刀**——那个数到底怎么算。

把"早单率"拆开看:

- "按地区拆"、"只看 Q4"、"排除测试数据"——这些是**维度和过滤**,提问时说清楚就定了,不欠定。
- "要不要 JOIN 产品表 / 区域表"——这是**物理路径**,数据库 schema 加上业务关系图就定了,不欠定。
- **"早单 = `status='CONFIRMED'` 还是 `IN ('CONFIRMED','SHIPPED')`"——这才是欠定的那一刀。** 它是**度量口径**问题,答案不在数据里,在组织共识里。

整个赛道这三年的所有崩溃,根因都在这最后一刀:**LLM 被迫去猜一个只存在于人脑里的口径,然后自信地把猜测当事实报出来**。

---

## 二、我的观点

一句话,本文要说服你的就这一句:

> **数据分析需要的不是更准的 AI,是它的编译器。**
> **这个编译器的"类型系统"是本体,"求值规则"是口径——两者缺一不可,且相互缠绕。**

把这句话从中间剖开:
- "本体"管"**什么东西、之间什么关系、有哪些属性、各自叫什么名字**"——它是这个领域的类型系统。
- "口径"管"**一个具体的业务数字到底怎么定义**"——它是这个领域的求值规则。
- 把两者咬合在一起,你才得到一个**编译器**:良构的问题确定地编译成 SQL,病构的问题被报错拒绝,而不是被错编译。

后面三节讲清这句话:为什么这个方向是反直觉但正确的(§三)、本体和口径具体是什么以及怎么缠绕(§四)、它实际上改变了什么(§五)、它在哪里会坏(§六)。

---

## 三、为什么这不是显然的

如果"造一个编译器"这么显然,为什么过去十几年没人做出来?

### 3.1 为什么 dbt / Cube / LookML 不是这个编译器

它们都做了"组织维护一组指标定义"——这其实是这个方向上最关键的一步。但它们停在了**语义层**,没有变成**编译器**。

差别在哪?在于**有没有那道闸门**。

在 dbt / Cube / LookML 的世界里,你可以查询定义好的指标,也可以**绕开它**:写一段自定义 SQL,跑一个临时聚合,没有任何东西阻止你产出一个"没有任何已授权口径背书"的数。LLM 接上来之后,这个问题被放大:LLM 永远倾向于"答得出来",哪怕代价是拼一个没人背书的 SQL。

编译器的本质从来不是"能生成代码",而是:

> **良构的程序编译,病构的程序被报错拒绝——而不是被悄悄错编译。**

少了"病构必须报错"这一闸,语义层永远是语义层,不会变成编译器。后面 §四会讲这道闸的具体名字。

### 3.2 为什么大模型公司不会顺手做掉这一层

OpenAI / Anthropic / Google 都会持续 ship "agentic data analyst"。但他们做不了**本体 + 口径**这一层,理由很简单:

口径必须由**业务方**写。"早单"是 `CONFIRMED` 还是 `CONFIRMED+SHIPPED`,不是技术问题,是组织共识问题。本体里"产品和早单之间走哪条 JOIN 路径",也不是技术问题,是业务问题。**这两件事天生不可由通用模型预训出来**——它们就是每家公司不一样的、需要被一条条人工写下来的东西。

大模型公司能做的是"更准的猜测"。但在"猜"这条路上,准确率从 90% 提到 95%、99%——失败模式不变,**你依然在使用一个无法被验证的黑盒**。这条路的天花板就是 99.x%,而 99.x% 在决策场景下依然是不可用的。

所以这一层不会被"上游做掉"——它是**新基础设施**,需要重新长出来。

### 3.3 为什么"准确率"是错的北极星

这是整个赛道最大的认知错误,也是最隐蔽的一个。

所有"AI 对话数据"产品都在优化**准确性**——更大的模型、更好的 prompt、更聪明的 RAG。但在一个没有裁判的领域,**"准确性"根本不是一个能被直接优化的目标**:你连"这一次对不对"都无法判定,又拿什么梯度去下降?

正确的北极星,应该是一个更朴素、也更要命的问题:

> **我怎么知道这个答案是错的?**

这里有一个反直觉、但关键的事实:

**一个你无法判断对错的"准确"答案,比一个你能指着说"这里错了"的、可能错的答案,更危险。**

LLM 报一个"12.3%",哪怕它 95% 的时候是对的,你也**无法在当下知道这一次是不是错的那 5%**——拿着它砍预算的人,赌的是运气。而一个**可以被指着定义、当场反驳**的数不一样:你能打开它背后那条口径,看见"早单 = `status='CONFIRMED'`",当场说"不对,我们还要算 SHIPPED",改那一行,所有未来查询同步修正。

借波普尔一句:**一个不可证伪的断言,对决策毫无价值**。LLM 的"12.3%"不可证伪——你根本不知道拿什么去验它。编译器编译出来的数可证伪——它锚在一条显式定义上,你能逐条质疑。

**编译器给数据分析的礼物,从来不是正确性,是可证伪性。**

这就让"一致性优先于正确性"那句话终于有了它的因果链:**正确性无法直接优化(没有裁判);但一致性可以——而一致性正是可审计的前提**。同一个问题永远得到同一个答案,你才能把那个答案钉死、追溯到那条定义、再反驳它;每次都飘的答案,根本无从审计。

## 四、解决方案:本体与口径的缠绕

到这里所有概念铺垫完毕。接下来是这套系统具体长什么样,以及为什么"本体"和"口径"这两件事必须**一起**出现——单独哪一个都不构成编译器。

### 4.1 本体是什么

**本体(ontology)** 不是 schema。Schema 告诉你数据库里"有什么列",本体告诉你"这些列在业务上意味着什么、它们之间真实的业务关系是什么"。

具体而言,本体由几件东西组成,**全部入库存在 Postgres 表里**,有 CRUD、有版本、有审计:

| 本体元素 | 表 | 业务含义 |
|---|---|---|
| **对象定义(OD)** | `ont_object_definition` | 一个业务实体,如 `EARLY_ORDER`、`PRODUCT`、`CUSTOMER` |
| **属性(Property)** | `ont_property` | OD 上的一个字段,有 display_name、aliases、type |
| **关系(Link)** | `ont_link` | 两个 OD 之间的、带类型的业务关系(哪个外键、几对几) |
| **关键词(Keyword)** | `lakehouse_keyword` | 用户可能在自然语言里说出来的词,挂到属性、值、OD 上 |

关键点:

1. **本体是人造的**,不是模型自学。是业务方协同 AI 在对话里一条一条建出来,然后入库存档。
2. **本体里所有东西都是有限集**:OD 总共就这么多张、Link 关系图是预先连好的、关键词词典是策展过的。
3. **LLM 永远在这些有限集里做选择**——挑哪个 OD、挑哪个属性、挑哪个关键词。它**不生成 SQL,不构造 JOIN,不发明字段名**。

这一点至关重要:它是后面"LLM 永远看不见 JOIN"的物质基础。

### 4.2 口径是什么

如果本体是"什么对象、什么属性、什么关系",那**口径(metric)** 就是"在这些对象之上,某个具体业务数字到底怎么定义"。

一条口径就是一条挂在某个 OD 上的、最精简的度量,被拆成三件事入库存在 `lakehouse_metric` 表里:

```sql
-- 在「早单」对象上,按订单类型给出订单数
select "ORDER_TYPE", sum("ORDER_QUANTITY") as "TOTAL"
from "EARLY_ORDER"
group by "ORDER_TYPE"
```

拆成:
- **主对象** = `EARLY_ORDER`(它必须是本体里已有的某个 OD)
- **度量** = `sum(ORDER_QUANTITY)`(必须是该 OD 上一个真实属性)
- **基础维度** = `ORDER_TYPE`(必须是该 OD 上一个真实属性)

注意它**只回答"算什么"**——不回答"怎么筛、按什么再拆"。那些都是运行时的事。口径本身**就是这个数在这家公司里的法定定义**,一条数据库行,可以被打开、可以被读、可以被反驳。

然后是那道关键的闸门:

> **任何一个聚合度量,必须被一条已授权口径背书。否则系统拒绝执行,返回 `NO_AUTHORIZED_METRIC`——而不是去拼一个没人背书的数糊弄过去。**

这一道闸,就是"语义层"和"编译器"的分水岭。它把整个系统的失败模式从"自信答错"挪到了"诚实拒答"——只剩两种结局:

> **要么答出口径定义的那个值,要么告诉你这个口径不存在。**

### 4.3 二者怎么咬合

到这里大多数读者会问:那本体和口径具体怎么协同?为什么必须**一起**?

四个层面:

**结构上:口径没有本体就没有家**。一条口径要存在,必须先有它挂的那个 OD;它的"基础维度"必须是这个 OD 上的一个真实属性;它的"度量"必须聚合在这个 OD 的某个属性上。没有本体,口径就是孤悬空中的一段 SQL,无法被组织、被复用、被关联。

**运行时:本体给路径,口径给聚合**。当用户问"各品牌的早单量是多少"——

1. 调度层(我们叫 SmartQuery 引擎)拿到自然语言。
2. LLM 在本体里挑:目标口径 = "早单量"(主对象 `EARLY_ORDER`)、维度关键词 = "品牌"(命中 `PRODUCT.BRAND` 属性)。
3. 引擎查本体的关系图:`EARLY_ORDER` 怎么连到 `PRODUCT`?——本体里 `ont_link` 表早就把这条 JOIN 路径预先连好了。
4. 引擎沿这条 Link 自动拼 SQL:`EARLY_ORDER` JOIN `PRODUCT` ON ... GROUP BY `PRODUCT.BRAND` 算 `sum("ORDER_QUANTITY")`。
5. 执行、返回。

**LLM 全程没碰过一行 SQL,没见过一个 JOIN**。它只做了两件**有限集里的选择**:挑哪条口径,填哪个关键词。SQL 是一个**确定性程序**沿本体关系图算出来的——给定同样的本体快照、同样的口径、同样的参数,产出永远是同一条 SQL、同一份结果。

**语义上:本体说"什么",口径说"哪个数"**。这两个问题在数据分析里都不平凡。本体回答"在我们组织里,'品牌'指 `PRODUCT.BRAND` 不是 `MARKETING_BRAND`""'早单'是个独立的对象,不是订单的子集";口径回答"'早单量'就是 `sum(ORDER_QUANTITY)`,不是 distinct count""'早单率'是早单量 / 总订单量"。把两者拆开,你才能让人类去审计每一条。

**对 LLM 来说,三件事都是有限集选择**:

- 挑 OD——本体里就这么多张;
- 挑口径——这个 OD 上已授权的口径就这么几条;
- 挑关键词——业务方策展的词典里就这么多。

每一件都来自数据库,都可以被打开看见、被讨论、被改。LLM **永远在做选择题,不做填空题**。

这就是为什么"本体 + 口径"才构成编译器,缺一不可:
- **只有本体没有口径**——你只知道"有早单这个对象",但每次"早单量是多少"还是要 LLM 自己拼聚合。回到 Text2SQL 的失败模式。
- **只有口径没有本体**——你只能查"在这一张表里算这一个数",一旦问题跨对象(品牌、地区、渠道)就需要 JOIN,JOIN 路径没有处描述,LLM 再次开始猜。

只有两者一起,LLM 的工作才被压缩到**纯粹的选择题**,而 SQL 的生成才能完全交给确定性程序。

### 4.4 一条口径,回答很多问法

编译器不能是僵的。同一条"早单量"口径,要能回答"早单量是多少""按品牌的早单量""Legion 品牌的早单量""按品牌看 Legion 早单量"——只能改一条口径就累死人。

它的灵活来自对每个属性的**运行时三态**:

- **不传** → 该列完全不出现;
- **传了、值留空** → 仅作维度展示(`group by`,不过滤);
- **传了、且有值** → 过滤 + 展示("过滤即展示")。

于是一条"早单量"口径,一字不改,就能由 LLM 通过三态填参覆盖上面所有问法——包括沿本体 Link 自动 JOIN 到产品对象的那些。

### 4.5 自然的延伸:口径依赖图

有了编译器,下一步自然是**让口径之间相互依赖**。

把每条口径看成一个**节点**;一条**有向边**表示"口径 B 要先拿到口径 A 的产出才能算"——边本身承载传参绑定:A 的某个输出列,绑定到 B 的某个参数。

一旦有了这张图,一个根本性的转变发生了:**面对一个问题,我们不再问"它有几个意图、该拆成几个子问题"**——那是过去最脆弱、最靠 LLM 猜的一步。我们只问:这个问题命中哪条**目标口径**?它的参数,哪些由问题直接给、哪些要由别的口径的产出来喂?

"任务拆解"不再是 LLM 的一次推理,而是**对一张策展好的口径图做确定性求值**:从目标口径反向链,逐个把未满足的参数接到能产出对应类型的上游口径,直到所有叶子参数都能由问题本身提供。这跟 dbt 的 `ref()` 依赖图、类型化函数组合、Datalog 反向链是同一个形状。

但有一条线必须守住,否则它会塌回成一个开放搜索、没有边界的"会规划的 AI"——这套范式存在的全部意义,就在于避开它:

> **依赖边必须是人策展的、带类型的;LLM 只负责选目标口径 + 填叶子参数;执行器是确定性求值器,不是开放搜索的 planner。**

守住这条线,"要么答对、要么说不存在"就从单条口径推广到整张图。

## 五、效果:你的失败模式变了

这一节最该写"已经被谁部署在哪",但我得诚实——出于信息安全,具体名字暂时不便披露。所以这一节不讲"谁在用",讲**你用了之后,失败模式具体变成了什么样**。

这是更扎实的效果陈述,因为它直接对应你日常工作中那个最痛的瞬间:**当 AI 给你一个数,你怎么办**。

| 维度 | Text2SQL(LLM 直接拼 SQL) | text2ontology(本体 + 口径) |
|---|---|---|
| 错误空间 | 无界。任何看起来合理的 SQL | 有界。只能从 `aliases[]` ∪ 已授权口径 ∪ 已策展 JOIN 路径这个有限集出错 |
| 错误定位 | 不可定位(LLM 黑盒) | 可精确定位:日志会告诉你召回走的是 EXACT/FUZZY/VEC 哪层、命中哪个 keyword id、用的哪条口径 |
| 错误修复 | 改 prompt → 影响未知 | 改一行 DB → 影响范围明确,所有未来查询一并修复 |
| 错误重现 | 不可重现(LLM 非确定性) | 完全可重现(本体快照固定即 SQL 固定) |
| 错误审计 | 决策时无法证明"为什么相信" | 决策时可指着口径行说"这个定义业务方 Q3 签过字" |

最直接的对比是这样的:在 Text2SQL 路线下,你拿到一个 12.3%,你能做的事是"再问一遍看看是不是一样"——而它每次都可能给你不一样的数。在编译器路线下,你拿到一个 12.3%,你能做的事是**点开这个数背后那一条口径,看见"早单 = `status='CONFIRMED'`",当场说"不对,我们还要算 SHIPPED",改那一行,所有未来查询同步修正**。

**前者是赌运气;后者是 5 秒钟内能在数据库里反驳一个数**。

这才是这套系统真正的卖点。它从来不卖"更准"——它卖"**当它错的时候,你立刻知道、能定位、能一次修好**"。在一个没有真理裁判的领域,这件事比"偶尔蒙对"值钱得多。

## 六、问题与边界

把刀口朝向自己,这套系统也不是没有边界:

1. **编译器检查一致性,不检查真相**。授权了的错口径,照样被忠实编译。系统保证"答案和口径一致、错误有地址",**不保证你的口径在业务上是对的**——那永远是人的责任。这是一个特性而非 bug:它把"判断对错"这件事**显式地**留在人的位置上,而不是假装机器能做。
2. **新的失败模式是"假不存在"**。口径其实有,但分词 / 召回没命中,于是误拒。错从"自信答错"挪到了"该答却拒答"——这是**严格更好**的失败(诚实 > 错答),但它把担子全压到了分词和召回的质量上。**分词和关键词召回依然是 60% 以上成败的命门**。
3. **派生度量和集合值是确定性最薄的地方**。比值(早单率 = 早单 / 总量)、要不要对上游产出的集合做迭代(map)——口径依赖图在这块会从"简单 DAG"升级成"带 map/reduce 的小型数据流语言"。这是最难、最值得继续打磨的部分,也是 v0.x 路线图的重点。
4. **冷启动时编译器是空的**。口径集为空、关系图未连 → 一切都拒答。这台编译器的"文法"是业务本体工程师一条条写出来的。**革命性在范式,兑现靠策展**——这是一个对组织肌肉而非对模型能力的要求。

四件事都没解决,但它们都**不是黑盒问题**——每一条都可以被定位、被讨论、被分摊到具体的人头上去推进。这本身就和"训出一个更准的模型"是两个物种的事。

---

## 七、结语

写代码的人花了几十年,才让"编译器"这三个字成为理所当然。一个程序员不会问"我的 GCC 这次会不会给我个不一样的结果"——因为编译器就是确定的、可重现的、可调试的。

数据分析没有这个。三年了,所有"AI 对话数据"产品给你的依然是**一个会变的数,你没法验它、没法争它、没法在它错的时候找到错在哪**。

这篇文章想说的是:这件事的解法不是"让 AI 更准",是给数据分析造它的编译器——**用本体管住"什么对象、什么关系、什么属性"(类型系统),用口径管住"每个业务数字怎么定义"(求值规则),让 LLM 只在这两个有限集里做选择,SQL 由确定性程序沿本体路径编译出来**。

代价是诚实的:它编译的是**与组织共识的一致性**,不一定是**客观真相**。但这恰恰是它最该被理解的地方——**在一个没有真理裁判的领域,能知道自己什么时候错、错在哪、怎么一次修好,比偶尔蒙对值钱得多**。

工程实现已开源,可读可批可复用:

→ **[text2ontology](https://github.com/agentofreef/text2ontology)** (Apache 2.0)

七服务 Postgres-only 架构、四层 hexagonal、CI 强制 import 单向、agent 决策行级日志可重放。代码里直接可以看到本节讲到的所有东西:`ont_object_definition`、`ont_property`、`ont_link`、`lakehouse_metric`、`lakehouse_keyword`、SmartQuery 引擎、`NO_AUTHORIZED_METRIC` 这道闸。欢迎 fork、批评、提 issue 反驳。
