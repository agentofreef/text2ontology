# 业务本体工程师:一个即将出现的新职业角色

> Business Ontology Engineer (BOE) · AI 时代,谁让企业的"数据真相"成立

---

过去 18 个月,我们见证了 **Prompt Engineer** 从一个被嘲笑的虚构职位,变成 LinkedIn 上写在 JD 顶端的真职位。同样的事情即将发生在另一个角色上:**业务本体工程师 (Business Ontology Engineer, BOE)**。

这个角色已经存在 —— 在过去 5 年的某些 AI 落地项目里,某个人事实上扮演了这个角色,只是他的头衔可能是 Data Analyst、Business Analyst 或 Solution Architect。**他们做的事情其实是 BOE 的工作,但还没有人这样命名**。

本文做三件事:
1. **先讲清楚本体到底是什么、不是什么** —— 防止"本体"变成下一个"Agentic"那样被市场掏空意义的空壳词
2. 然后讲 BOE 这个角色具体做什么、跟现有角色(尤其是 FDE)的差别
3. 最后给数据分析师、CTO、教育机构三种人的启示

---

## 一、先讲清楚:本体是什么

本体 (Ontology) 在企业语境下,**有一组可操作的定义**。如果你听到有人说"本体",你可以用以下五条标准检查他说的是不是真的本体:

### 1. 本体是**一组业务概念 + 它们之间关系**的结构化表达

具体一点:
- "订单"是一个业务概念
- "客户"是一个业务概念
- "订单属于客户"是它们之间的一种关系
- "订单"有"金额"、"状态"、"下单时间"等属性
- "金额"是数值,"状态"是机器码,"下单时间"是时间戳

把这些概念、属性、关系**用 DB 表表达出来**,就是本体。在 [text2ontology](https://github.com/agentofreef/text2ontology) 的实现里:
- `ont_object_type` 表存对象类型(订单、客户、产品)
- `ont_property` 表存每个对象的属性
- `ont_link_type` 表存对象间的链接(订单 → 客户)

### 2. 本体是**业务术语 → 数据列的双向映射**

用户说"早单率",这个词:
- 映射到 `ont_object_type='Order'`
- 加上 filter:`ont_property='status' WHERE value IN ('CONFIRMED')`
- 加上聚合:`COUNT(*) / total_orders`

这个映射不是 prompt 里描述的,是**写在 `lakehouse_keyword` 表 + `lakehouse_metric_intent` 表里的结构化数据**。

### 3. 本体是**可被 CRUD、可被审计、可被版本化的资产**

- 任何人可以 SELECT 看本体当前状态
- 任何变更走 INSERT/UPDATE,留下记录(who, when, why)
- 出错时可以回滚到任意时间点
- 这就是为什么本体能成为 [真理裁判](./manifesto.zh.md) —— 它有事实的所有性质

### 4. 本体是**人造的**,不是算法抽取的

这点很重要 —— **本体必须有人为它签字**。
- AI 可以**提议**新的对象、新的属性、新的链接
- 但本体的状态变更必须经过**业务方签字**
- 这就是为什么有了 BOE 这个角色 —— 必须有人**负责签字**

### 5. 本体是**特定业务场景的客观对象集合**

强调"特定业务场景":本体不是通用的、跨行业的。**做电商的"订单"和做 SaaS 的"订阅"是不同的本体**。本体跟着业务走,不是跨业务复用的。

强调"客观对象":本体描述的是**这家公司业务里实际存在的东西**,不是理论上可能存在的东西。如果你的公司没有"经销商"这个业务概念,你的本体里就不该有 `Distributor` 对象。

## 二、本体不是什么(防 hype 的核心防线)

我们刚刚解构了 ["Agentic" 这个词的 5 层错觉](./ai-agentic-illusion.zh.md)。**我们不能让"本体"重蹈覆辙**。所以下面这 7 条要逐条说清:

### 不是 schema

Schema(数据库表结构)是**物理事实**:这张表有哪些列、什么类型、什么约束。
本体是**业务事实**:这些列对业务意味着什么、谁在乎、什么时候用。
**两者描述同一份数据的两个不同层面**,缺一不可。Schema 给数据存在性,本体给数据意义。

### 不是知识图谱 (Knowledge Graph)

Knowledge Graph 是**通用领域知识网络**(Wikidata、Google KG)。
本体在我们的语境里是**单个企业的业务事实层**。规模和目的都不一样。**KG 关心"鲍勃·迪伦是谁",本体关心"我们公司里的'活跃客户'到底指哪些人"。**

### 不是 OWL / RDF / Semantic Web

OWL / RDF 是 W3C 2000 年代推动的语义网标准。它们是**学术 + 标准化**的本体表达。我们用的本体是**工程实践**:用 Postgres 表实现,用 SQL 操作,用现代 LLM 协作维护。**血统相同,工具完全不同**。OWL/RDF 没有大规模商业成功,我们的本体走的是 SaaS 工程路径。

### 不是 LLM 的 Prompt Context

把 schema 描述塞到 prompt 里、把 metric 定义写进 system message 里 —— 这是**用 prompt 模拟本体**,不是本体。
- Prompt 不可版本化
- Prompt 不可审计
- Prompt 不可被多个系统共享
- Prompt 不可被业务方 CRUD

**Prompt 是 LLM 的私有记忆。本体是企业的公共资产。**

### 不是 Vector Database

Vector DB 是**相似度检索基础设施**。它存的是 embedding,做的是 nearest neighbor 查询。
本体是**断言式的事实声明**。它存的是结构化关系,做的是精确查询。
**它们经常一起用,但角色完全不同**。Vector 帮 LLM 在大语料里找相关片段,本体直接告诉 LLM "这个词在我们公司就这么定义,没有相似度,没有近似"。

### 不是 Data Dictionary

Data Dictionary 是**列级元数据表**:某列叫什么名字、是什么类型、是谁建的。
本体是**业务概念层**:某概念跨多张表、跨多个列,有自己的生命周期。
**Data Dictionary 是给数据库管理员看的,本体是给业务方看的**。

### 不是 dbt Semantic Layer / Cube.js

dbt Metrics 和 Cube 是 metric-centric 的 semantic layer —— 它们让你定义 metric(指标)的 SQL 模板。
本体**包含**这层,但更大:本体还包括对象、属性、关系、别名、因果链、学习事实。**Metric 是本体的一个子集,不是全部**。

### 一个简洁判断标准

如果你听到有人说"我们做了本体",问他四个问题:

> 1. 你的本体存在哪张 DB 表里?(回答不出 = 不是本体,是描述)
> 2. 我能不能 INSERT 一个新对象?(回答不出 = 不是本体,是固化的代码)
> 3. 上一次有人改本体是什么时候,改了什么?(回答不出 = 没有审计 = 不是本体)
> 4. 如果我改一个对象的定义,所有用它的查询会跟着变吗?(回答 no = 不是本体,是文档)

**四个问题都能答出来才是本体。其中任何一个答不出,只是 RAG / prompt / 文档 / glossary 在借本体的名字**。

## 三、业务本体工程师 (BOE) 具体做什么

有了上面的定义清晰,BOE 的工作内容就具体了。一个 BOE 的日常职责:

1. **跟业务部门访谈**,把口头业务术语提炼成结构化对象、属性、关系
2. **维护本体表**(`ont_object_type`、`ont_property`、`ont_link_type`、`lakehouse_metric_intent`、`lakehouse_keyword`),增删改查
3. **审查 AI 提议的本体变更** —— 当 Agent (Builder 模式) 提议新建一个对象,BOE 决定通过 / 修改 / 拒绝
4. **监督 LLM 分词输出**,矫正错误(系统每跑一次查询,BOE 都应该看一眼分词是不是合理,需要时补 alias)
5. **跟数据工程师对接 staging schema 变化** —— 上游表结构变了(列名改了、表迁移了、数据源换了),BOE **只更新本体的"实现层"**:对应 OD 的 `semantic_sql` 字段、`source_column` 映射。**本体的业务层(对象定义、属性语义、关系、业务口径)完全不动**。这是本体存在的根本价值之一 —— **让业务语义免疫于物理 schema 的频繁变动**
6. **跟业务决策者签字业务口径** —— "早单"到底怎么定义,书面落地
7. **配置和审查 L3 验证器** —— 系统在 ship 一个本体变更前要跑 5 个验证器,BOE 看结果
8. **处理边界 case** —— 当系统遇到模糊问题(歧义 token 没法消除),BOE 介入做决策
9. **设计数据治理流程** —— 哪些业务变更必须经过本体,哪些可以走快通道

跟 FDE (Forward Deployed Engineer) 的关键对照:

| 工作场景 | FDE(供应商派的) | BOE(企业自己的) |
|---|---|---|
| 业务术语澄清 | 项目期一次性访谈提炼 | 永久持续,每周迭代 |
| Schema 改动 | 不主动动,只用既有数据 | 主动跟数据团队对接 |
| AI 输出验证 | 项目期内做,合同结束就停 | 永久职责 |
| 责任承担 | 合同期内,跟供应商共担 | 永久,企业内承担 |
| 离职 / 项目结束 | 业务知识带走,客户重新学 | 知识沉到本体表,留下 |
| 报酬来源 | 客户付给供应商,供应商付 FDE | 企业直接雇佣 |
| 利益最大化方向 | 供应商续约 + 扩大账户 | 客户自己的数据可信度 |

**FDE 是租来的责任承担者。BOE 是自己培养的责任承担者。**

## 四、能力画像:四角混合

一个合格的 BOE 是四种现有职业能力的合成:

- **30% Business Analyst** —— 能跟业务部门访谈,理解他们说"运营效率"时到底是什么
- **30% Data Engineer** —— 能写 SQL,能读 Postgres execution plan,能跟 staging schema 打交道
- **30% Knowledge Engineer** —— 懂概念建模、关系建模、本体设计的方法论(虽然不必懂 OWL)
- **10% AI 协作员** —— 知道何时让 AI 提议、何时挑战 AI 提议、何时不让 AI 介入

不是上面任何一种角色的 1 倍,是这四种的 0.3+0.3+0.3+0.1 加权混合。**这就是为什么没有现成的人能直接上岗 —— 没有学校训练这种混合,只能从已有职业里转**。

## 五、为什么 BOE ≠ 现有任何角色

- **不是 Business Analyst (BA)**:BA 不动 schema,不写 SQL,不审 AI 输出
- **不是 Data Analyst (DA)**:DA 回答具体业务问题,不维护语义基础设施
- **不是 Data Engineer (DE)**:DE 建 pipeline,不维护业务口径
- **不是 Knowledge Engineer (KE)**:传统 KE 在 1980s-90s 的专家系统时代,不跟 LLM 协作
- **不是 Prompt Engineer (PE)**:PE 优化 prompt 文本,跟数据治理无关
- **不是 ML Engineer (MLE)**:MLE 部署模型,BOE 部署业务意义

### 特别地:不是 FDE

FDE 是市场上最像 BOE 的角色。两者都是"业务 + 技术混合 + AI 时代专属"。但它们**站在不同战场的两侧**:

- **FDE 替供应商工作**,目标是供应商的续约 + 扩大账户
- **BOE 替客户工作**,目标是客户自己数据真相的完整性

- **FDE 是过渡形态**:在企业还没有 BOE 的时代,只能租 FDE 应急
- **BOE 是终态**:企业建立了自己的本体能力后,可以摆脱对 FDE 的依赖

> 过去 18 个月企业大量招 FDE,是因为他们没有第二个选项。
> 当 [text2ontology](https://github.com/agentofreef/text2ontology) 这类基础设施成熟后,**越来越多企业会把"租 FDE"的预算转向"招 BOE"**。

这不是说 FDE 会消失。FDE 会继续做**项目启动期 + 复杂集成期**的工作。但**日常本体维护、口径审查、AI 输出验证这些永久工作,会被 BOE 接走**。

## 六、在哪种公司率先出现

第一批 BOE 招聘会出现在三类企业:

1. **数据治理重的行业** —— 金融、医疗、制药、保险。这些行业的合规要求**强制需要可审计的口径管理**,本体不是可选项
2. **中型企业(500-5000 人)**—— 大企业能买 Palantir 派 FDE,小企业雇不起任何东西。中型企业既有需求又有预算,但请不起 Palantir 级别的合作,**自己培养 BOE 是最经济的选项**
3. **AI-native startup** —— 从 day 1 就把本体当核心架构资产建起来,而不是 retrofit。这些公司的早期 BOE 经常是 founder 自己

## 七、量级与时间线预测

- **2026 年底**:全球 BOE 实际担当此职责的人 < 1000,且大多数没有这个 title(挂在 BA / DA / SA 名下)
- **2027-2028**:第一批以 "Business Ontology Engineer" 为正式 title 的 JD 出现,招聘市场开始认这个词。规模类似 Prompt Engineer 在 2023 上半年的早期阶段
- **2029-2030**:大企业标配,北美 P75 薪资将与 ML Engineer 接近($180-250k base),原因是混合技能稀缺
- **2032+**:本科 / 硕士项目开始开"业务本体工程"或类似方向,**第一所开设此专业的院校会复制 2015 年的 Data Science 项目盛况**

## 八、给三类人的启示

### 给数据分析师 / 业务分析师 / Solution Architect

如果你已经在企业里做"理解业务术语 → 翻译成数据查询"这件事,**你的 5 年内最自然的转型方向就是 BOE**。三件事可以开始:

1. **学结构化本体建模**:不需要学 OWL/RDF,直接看 [text2ontology spec](https://github.com/agentofreef/text2ontology) 这类工程化实现
2. **学 LLM 协作**:用 Cursor / Claude / GPT 帮你提议本体变更,练习"批改 AI 的提议"
3. **在你当前的项目里,选 1 个业务术语,自己手动建一个 mini ontology**:存到 Notion / Excel / Postgres 都行,但要 follow 上文 5 条本体定义

### 给企业 CTO / Chief Data Officer

你下一个 hire 不是 Data Scientist,是 BOE。

**因为**:你的 AI 部署目前最大的瓶颈不是模型能力,是"AI 输出谁来背书"。BOE 是那个背书的人 —— 通过维护本体让 AI 输出**变成可审计的、deterministic 的、可问责的**。

招 BOE 的 ROI 比招 Data Scientist 高,**因为前者解决责任问题,后者只解决能力问题**。能力没有责任承担时,在企业是 100% 用不起来的。

### 给学校 / 培训机构 / Bootcamp

**第一所开"业务本体工程"专业的学校会赢**。

需要的课程组合:
- 数据库与 SQL(已有)
- 业务流程建模(已有,在 BA 系)
- 本体论与知识表达(已有,在哲学/CS 系,但很冷)
- LLM 协作与提示工程(已有,新)
- 企业数据治理(已有,在 MIS 系)

**把这五门课打包,加一个 capstone project 让学生在真实企业部署一套 mini ontology** —— 这就是 BOE 项目的最小教学单元。

## 收尾:命名先于现实,但通常晚 5 年

一个新职业的诞生,**通常先于这个职业的命名 5 年**:
- 1990s 末的"网页设计师"在 1995 年就在做事,2000 年才被市场承认
- 2008 年的"数据科学家"在 2002 年的对冲基金里已经存在,只是叫"quantitative analyst"
- 2015 年的"DevOps 工程师"在 2010 年的初创公司里已经存在,只是没人这样叫
- 2023 年的"Prompt Engineer"在 2020 年的 AI lab 里就在做事

**业务本体工程师在 2024-2025 年就在做事,只是分散挂在 BA / DA / SA / FDE 这些既有头衔下**。我们用这篇文章给这个角色发一张命名证书。

**5 年后,招聘 JD 上会有这个 title。希望那时你能想起来:**
**第一份命名,在 2026 年这里诞生**。
