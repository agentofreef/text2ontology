# 本体先于查询

> 在 LLM 无法判断对错的领域,我们如何重建确定性

---

AI 已经在写代码。Cursor、Copilot、Devin 让一个人完成过去十人的工程产出。但在数据分析这个**比写代码大十倍的领域**,同样的 LLM 技术过去三年没能突破任何企业主流。

不是模型不够强。不是 prompt 没调好。是**范式错了**。

本宣言主张:**让 LLM 直接从自然语言写 SQL 的路线,在企业场景下结构性不可行**。出路是反过来 —— 先用 AI 协助把"业务到底在分析什么"固化成**本体**,再让查询从本体里长出来。

---

## 一、真理裁判的缺失

软件工程里有一个经典概念叫 **真理裁判** —— 一个能告诉你"正确答案是什么"的黑盒。AI 编程之所以能闭环收敛,是因为**测试就是真理裁判**:模型产出代码,跑测试,通过或失败,迭代。

**数据分析里没有这样的裁判。**

你问"早单率是多少?",LLM 给出"12.3%"。你**没有任何自动机制**能验证这个 12.3% 是对的。"早单"在你公司的定义是 `status='CONFIRMED'`?还是 `status IN ('CONFIRMED', 'SHIPPED')`?这取决于人脑里的共识,**不存在于任何代码或数据里**。

没有裁判 = 没有迭代收敛 = 永远是 demo,不是产品。

这不是模型能力的问题。这是范式的问题。

---

## 二、三个一起作恶的失败

主流 Text2SQL / "AI 直接对话数据"路线在企业场景下有三个失败,**它们共享同一个根因**:

### 失败一(理论):没有裁判,LLM 不能收敛

如上所述。这是**数学层面**的限制,不是工程问题。再大的模型也救不了。

### 失败二(实践):语义元数据是中型企业的致命门槛

Snowflake Cortex、Databricks Genie、Cube、dbt + LookML —— 这些方案都要求企业**预先建好一套语义层**:业务术语词典、schema 描述、指标定义、实体关系。

构建这套元数据需要一支专门的数据工程团队投入数月。对 Fortune 500 来说是可承受的成本,**对 80% 的中型企业是致命门槛** —— 他们既没有团队也没有预算,而且没有类似先例证明值得这个投入。

### 失败三(实践):90% 准确率不是企业可用准确率

任何 Text2SQL 公司的市场材料都会告诉你"准确率 90%+"。

**90% 意味着 10 个问题里有 1 个错的。**

当 CFO 根据 AI 的回答说"Q4 我们应该砍 20% 营销预算",如果这个回答错了,**谁来承担后果?**

这就是过去三年"AI 直接对话数据" demo 满天飞、企业生产环境寥寥无几的真正原因。**90% 在娱乐场景下是惊艳,在决策场景下是不可用。**

---

## 三、提议:本体即真理裁判

**在没有测试裁判的领域,我们造一个语义裁判。这个裁判就是本体。**

本体是什么:
- **人造的**:由本体维护者协同 AI 在对话中构建,不预先存在
- **版本化的**:存在 Postgres 表里,有增删改查、有 mark 三态、有审计日志
- **确定性的**:给定查询参数,口径永远生成同一条 SQL
- **可审计的**:任何决策者可以打开 `lakehouse_metric` 表,亲眼看见"早单"在你公司的定义到底是什么

它不是 schema —— schema 是物理事实,本体是**业务事实**。
它不是传统语义层 —— 传统语义层假设你预先建好;本体是**AI 协助你逐步建出来**的。
它不是 prompt 描述 —— prompt 不可版本化、不可审计、不可增删改查。

**信任路径在这里发生迁移**:用户不再被要求"相信 AI",用户被要求**相信本体**。本体错了,改一行 DB,所有未来查询同时修复。AI 错了,你根本不知道哪里错了。

这是企业愿意为后者付费的真正原因 —— **信任有了归宿**。

**Text2SQL 死在多表,本体路线把多表整个抽走**。让 LLM 自由写 SQL 的方案在 3 张表以上准确率断崖,因为 LLM 要同时决定"用哪张表 + 怎么 JOIN + 怎么写过滤聚合",任何一步错就全错。本体路线让 LLM **只在已经预先连好的 OD 网络里挑 OD、挑口径、挑关键词** —— 三件事都是有限集里的选择,不是生成。SQL 由后面的 SmartQuery 引擎顺着 OD 之间的 Link 自动拼出来。**LLM 永远看不见 JOIN**。这才是"LLM 只是受约束的执行者"的工程兑现。

### 口径:把"裁判"二字坐实的那一层

本体是一整张网 —— 对象、关系、知识、触发词。但一次提问里**真正欠定的,从来只是那个数**:"早单 = 哪个状态"。把这个数管死的东西有一个精确的名字 —— **口径(Metric)**:在某个 OD 上、一条最精简的度量定义,就是这个数在你们组织里的法定定义。

系统的铁律是:**任何一个聚合度量,必须被一条已授权口径背书。否则拒绝执行(`NO_AUTHORIZED_METRIC`),而不是拼一个没人背书的数糊弄过去。** 于是只剩两种结局 —— 要么答出口径定义的那个值,要么告诉你这个口径不存在。**"自信地答错"这条路被结构性地消灭了。**

这让本体从"语义层"升级成了**编译器**:良构的问题(有口径背书)被确定地编译成 SQL,病构的问题被报错拒绝,而不是被悄悄错编译。**数据分析第一次有了它的类型系统。**

### 北极星不是准确,是可审计

但要讲清楚它**编译的到底是什么**,必须先纠正一个整个赛道都搞错的北极星。

所有"AI 对话数据"产品都在优化**准确性**。可在一个没有裁判的领域,准确性根本不是能被直接优化的目标 —— 你连"这一次对不对"都无法判定。所以这套系统的北极星从第一天起就不是准确,而是 **可审计** —— 一个更朴素、也更要命的问题:**我怎么知道这个答案是错的?**

一个你无法判断对错的"准确"答案,比一个你能指着说"这里错了"的、可能错的答案,**更危险**。LLM 报一个"12.3%",哪怕 95% 的时候对,你也无法在当下知道这一次是不是那错的 5%。口径背书的数不一样 —— 它**指向一条你能打开、能读、能反驳的口径**:看见"早单 = `status='CONFIRMED'`",你当场能说"不对,我们还算 SHIPPED",改那一行,所有未来查询同步修正。

借波普尔一句:**一个不可证伪的断言,对决策毫无价值。** 编译器给数据分析的礼物从来不是正确性,是**可证伪性**。所以才有"一致性优先于正确性":正确性无法直接优化,但一致性可以 —— 而一致性正是可审计的前提。

> 这一节是宣言的核心,完整论证见 [《数据分析的编译器》](../essays/data-analysis-compiler.zh.md)。

## 四、九条原则

> 本宣言不要求你全部认同。如果你只接受其中一条并在你的项目里实践它,就已经走在了我们这边。

### 1. 无裁判,无收敛

没有真理裁判就别幻想 LLM 能收敛。在数据分析里,裁判必须人造 —— 这是工程问题,不是模型问题。

### 2. 业务语义是资产,不是 prompt

"早单"的定义是企业资产,不是 system prompt 里的一段描述。**资产入库,prompt 易腐。**

### 3. 查询从本体长出来,不从 schema 长出来

Schema 告诉你"有什么列",本体告诉你"这些列意味着什么"。把查询的起点放在本体而不是 schema,**这是范式分水岭**。

### 4. 与 AI 共建本体,而非先建后用

传统语义层要求企业先投入数月建元数据。我们拒绝这个前提 —— 用 AI 协助本体的**渐进式构建**,让中型企业也能进场。

### 5. 积累即复利:每次问答都是教学,不是消费

不要指望第一次提问就拿到正确答案。本体系统的智能是**积累出来的**,不是模型给定的。

每一次用户提问,本体维护者都应该检查 LLM 的分词输出 —— 补遗漏的关键词、修正错误的 alias、调整口径优先级。每次矫正都让系统下次更准。

**不做矫正,系统永远停留在冷启动。做了矫正,错误率每周复利下降。**

这是 Text2SQL 路线**根本不存在**的优势:它没有可矫正的层。每次查询是一次性的、不留痕迹的。我们的每次查询都在改造本体本身。

**销售视角的一句话**:
> 买 Text2SQL,你拿到的是当下的一个准确率数字。
> 买 Text2Ontology,你拿到的是一条不断上升的复利曲线。

### 6. 信任本体,不信任 AI

信任迁移是企业可付费的根本。让 AI 做确定性路径上的执行者,**让人类负责本体这层事实基底**。

### 7. 本体的每次变更必须可审计

AI 改本体必须走 草稿 → 验证 → 上线 流程。强制审计,强制人类签字。**没有这一层,所有"AI 改 schema"都是赌博。**

### 8. 记忆是结构化的,不是 token 堆

Agent 多轮对话的记忆不应该是历史 token 的堆叠,应该是结构化的"本轮已确认的本体单元"沉淀。**这是生产级 Agent 与 demo Agent 的分界线。**

### 9. 有界错误与无界幻觉是不同物种

我们不主张本系统不会错。我们主张本系统的**错误模型与 LLM 幻觉是不同物种**:

| 失误维度 | Text2SQL(LLM 直接拼 SQL) | text2ontology(本体路径) |
|---|---|---|
| 错误空间 | 无界。任何看起来合理的 SQL | 有界。只能从 `aliases[]` ∪ `口径.priority` ∪ `causality(join_key)` 这个有限集出错 |
| 错误定位 | 不可定位(LLM 黑盒) | 可精确定位:`agent_step` 表告诉你走的是 EXACT/FUZZY/VEC 哪层、命中哪个 keyword id |
| 错误修复 | 改 prompt → 影响未知 | 改一行 DB → 影响范围明确,所有未来查询一并修复 |
| 错误重现 | 不可重现(LLM 非确定性) | 完全可重现(本体快照固定即 SQL 固定) |
| 错误审计 | 决策时无法证明"为什么相信" | 决策时可指着口径行说"这个口径 CFO Q3 签过字" |

**有界错误是企业可接受的错误。无界幻觉不是。**

## 五、工程证据

[text2ontology](https://github.com/agentofreef/text2ontology) 是本宣言的工程实现。

### 架构

四层架构,每一层对应宣言里的一组原则:

```
┌────────────────────────────────────────────┐
│  AI Agent(查询 / 建模 / 治理)             │   ← 原则 4, 5, 7, 8
├────────────────────────────────────────────┤
│  转换引擎(本体 → SQL,确定性生成)         │   ← 原则 3, 6
├────────────────────────────────────────────┤
│  数据湖仓(Postgres + 本体语义层)          │   ← 原则 2, 3
├────────────────────────────────────────────┤
│  数据连接器(PBIT / Excel / CSV / PG 镜像) │   ← 原则 4
└────────────────────────────────────────────┘
```

四层导入方向严格单向,由 CI 强制。完整架构见 [docs/spec/architecture-overview.md](https://github.com/agentofreef/text2ontology/blob/main/docs/spec/architecture-overview.md)。

### 已实现

- **七服务 Postgres-only 架构**,四层 hexagonal,CI 强制 import 单向
- **两种 Agent 模式**:查询 / 建模,各自独立工具集和数据落点
- **五个 L3 验证器**:AI 改本体必须全绿才能上线
- **口径系统**:触发词 → 完整查询模板(度量 / 过滤 / 自动 groupBy / pivot),零代码新增
- **线程记忆账本**:跨轮结构化记忆,token 浪费降低一个数量级
- **agent_step 行级日志**:每一步 Agent 决策可重放

代码全部开源(Apache 2.0),欢迎 fork、批评、复用。

### 仍在路上(诚实状态)

**原则 7"可审计"目前是半完成态。** Agent 决策侧的日志(`agent_step`)、builder 模式的"propose → 人工 activate"流程已实现,但**本体本身的完整版本控制 —— 谁在什么时候改了哪条口径的哪个字段、改前是什么、能否回滚 —— 这一层还没写**。这是 v0.2 路线图的第一优先级。

其他原则基本到位,但每条都有边界 case 待打磨。

**我们不假装系统完工。我们假装信念清晰、方向锁死、欢迎贡献。**
