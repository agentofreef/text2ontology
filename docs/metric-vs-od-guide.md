# 指标(Metric)快速上手:它是什么 · 和 OD 的区别 · 各管什么

> 一句话:**OD 负责"把数据洗干净、命名好、定义好关系";指标负责"在某个 OD 上算一个度量";具体筛什么、按什么拆、连哪些表,是运行时由提问决定的,不写进指标。**

本文面向第一次配置指标的人,5 分钟搞清楚边界:什么该写进指标、什么不该。

---

## 1. 三层心智模型

一条查询从下到上分三层,职责严格分开:

```
运行时(function call)  ── 动态:这次问题要筛什么、按什么拆、排序、limit
        ↓ 由 LLM 按提问填
指标 Metric            ── 静态:一个度量 + 基础维度(它自己的固定口径)
        ↓ 基于
OD(对象定义)          ── 静态:洗好命名好的虚拟表 + 表间关系 + 基线过滤
```

| 层 | 负责什么 | 静态/动态 | 例子 |
|---|---|---|---|
| **OD** | 列映射、字段变换(CASE/计算列)、对源表的 JOIN、**基线过滤**、**OD 之间的连接键(关系)** | 静态,所有消费者共享 | `EARLY_ORDER` = 把若干源表洗成一张含 `ORDER_TYPE / ORDER_QUANTITY / GEO / PRODUCT_NAME …` 的虚拟表 |
| **指标 Metric** | **一个聚合度量** + **基础维度**(这条指标天生就按它拆) | 静态,一条指标一个口径 | `早单数量` = `sum("ORDER_QUANTITY")`,基础维度 `ORDER_TYPE` |
| **运行时(function call)** | 这次提问要加哪些维度、过滤哪些值、排序、limit | 动态,每次提问不同 | "Legion 品牌的早单量" → 运行时加 `PRODUCT.PRODUCT_BRAND = Legion` |

---

## 2. 指标到底是什么

**一条指标 = 一个 OD 上的、最精简的核心度量 SQL**:

```sql
select "ORDER_TYPE", sum("ORDER_QUANTITY") as "TOTAL"
from "EARLY_ORDER"
group by "ORDER_TYPE"
```

就这么多。保存时后端自动把它拆成结构化的三部分:

- **主 OD** = `EARLY_ORDER`(从 `FROM` 推出)
- **度量** = `sum("ORDER_QUANTITY")`(SELECT 里的聚合;`as TOTAL` 别名会保留为结果列名)
- **基础维度** = `["ORDER_TYPE"]`(SELECT 里的非聚合列 = 这条指标永远会带的分组)

> 指标"描述它自己",**不描述任何一次具体查询会长什么样**。

---

## 3. 运行时怎么在指标之上拼装

提问时,LLM 选中这条指标,再传一组 `filters`。引擎自动:

1. 把过滤/维度列**并入** SELECT + GROUP BY(过滤即展示);
2. 如果某列属于**别的 OD**(如 `PRODUCT.PRODUCT_BRAND`),**自动 JOIN** 那个 OD(连接键来自本体关系 `ont_causality(join_key)`,在"属性图谱"页面建立);
3. 用防 fan-out 的稠密分组结构执行,`sum` 不会被 JOIN 撑大。

**同一条指标,三种问法 → 三种结果(无需改指标):**

| 提问形态 | 运行时传入 | 引擎产出 |
|---|---|---|
| "各订单类型的早单量" | (空) | 仅按基础维度 `ORDER_TYPE` |
| "**各品牌**的早单量"(纯展示,不筛) | `PRODUCT.PRODUCT_BRAND`(**无值**) | 自动 JOIN PRODUCT,按 `ORDER_TYPE, PRODUCT_BRAND` 拆,**无 WHERE** |
| "**Legion** 品牌的早单量"(筛+展示) | `PRODUCT.PRODUCT_BRAND = Legion` | JOIN PRODUCT,过滤 Legion,且按品牌展示 |

这就是 **"属性三态"**:
- **不传** → 该列完全不出现;
- **传了、值留空** → 仅作维度展示(`group by`,无过滤);
- **传了、且有值** → 过滤 + 展示。

## 4. ✅ 什么该写进指标

- **聚合度量**:`sum/avg/min/max/count/distinct_count(<列>)`。一条简单指标**只放一个**度量。
- **基础维度**:这条指标天生就该按它拆的列(如订单指标按 `ORDER_TYPE`)。可以 0 个(纯总量)、也可多个。
- **主 OD**:度量所在的那个 OD(`FROM` 单个 OD)。
- **触发词**:让 recall 能命中这条指标的业务关键词(必填 ≥1;否则后端拒绝保存)。触发词会写入 `lakehouse_keyword` 并可被召回。
- **度量别名**(可选):`as "TOTAL"` → 结果列就叫 `TOTAL`,否则引擎默认 `Total_<列>`。

## 5. ❌ 什么不该写进指标

| 不要写 | 应该放哪 | 为什么 |
|---|---|---|
| 运行时过滤(`GEO = 'US'`、`品牌 = Legion`) | 运行时 function call | 同一条指标要能回答"所有 GEO""某个 GEO""某几个 GEO",过滤是提问决定的 |
| 可选/临时的拆分维度 | 运行时 function call | 用户这次想按品牌、下次想按机型,写死就僵了 |
| `JOIN`、跨 OD 的列写进 SQL | 引擎按 OD 关系自动 JOIN | 连接键是 OD 层的关系元数据,不是指标的事 |
| **基线过滤**(如"只看本财年""排除测试数据"——所有查询都该有的) | **OD 的 canonical_query** | 放 OD 里,所有指标自动继承;放指标里要每条重复 |
| `count(*)` | 用 `count(<某列>)` | JOIN 下 `count(*)` 会重复计数 |
| 多个聚合 / 嵌套 / 窗口函数 | 走遗留 SQL 逃生通道(`level='sql'` + `{sys.req/opt.NAME}`) | 简单模型只接单层、单度量;复杂形状用高级路径 |

> 经验法则:**问自己"这个东西是这条指标永远成立的,还是随提问变化的?"** 永远成立 → 进指标(或更基础的进 OD);随提问变化 → 留给运行时。

## 6. 怎么建一条指标(编辑页)

`/lakehouse/zh/ontology/lakehouse-metrics/new`:

1. **基本信息**:Name、显示名、描述、优先级、Mark。
2. **主 OD**:单选(度量所在的 OD)。
3. **指标 SQL**:写那条最精简的核心度量,例:
   ```sql
   select "ORDER_TYPE", sum("ORDER_QUANTITY") as "TOTAL"
   from "EARLY_ORDER" group by "ORDER_TYPE"
   ```
   —— 无 token、无 JOIN、无过滤。保存时自动解析。
4. **触发词**:加业务关键词(回车添加),至少 1 个。保存后写入关键词库、可被召回。
5. 保存。

> 复杂指标(嵌套/窗口/多度量)切到遗留 SQL 模式(`level='sql'`),用 `{sys.req.X}` / `{sys.opt.X}` 内联参数——但新模型的默认就是上面这条单层路径,优先用它。

## 7. 怎么测一条指标(函数调用模拟器)

编辑页顶栏 **「打开模拟器」** → `…/lakehouse-metrics/simulate?id=<id>`:

- 锁定 `intent`(指标名)、`odName`(主 OD)、度量(只读)。
- **属性行**:选 OD + 属性,**留空值=仅展示该维度,填值=过滤**;选别的 OD 的属性会自动 JOIN。
- **排序 / limit**;**表单 ⇄ JSON 切换**看真实 function-call 传参。
- **模拟运行** → 看引擎实际拼出的 SQL + 结果,和线上 LLM 走的是同一条拼装路径。

## 8. 一张速查表

```
OD        = 数据 + 命名 + 关系 + 基线过滤        (静态,所有指标共享)
Metric    = 一个度量 + 基础维度                  (静态,一条指标一个口径)
Function  = 这次要加的维度 / 过滤值 / 排序 / limit (动态,每次提问)
call

指标里只放:度量、基础维度、主 OD、触发词、(度量别名)
指标里不放:运行时过滤、可选维度、JOIN、基线过滤、count(*)、多/嵌套/窗口聚合
```
