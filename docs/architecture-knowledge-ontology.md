# 知识本体架构设计

## 1. 双本体模型

系统存在两种本体，服务于不同目的：

| | 数据本体 (Od) | 知识本体 (Ok) |
|--|--------------|--------------|
| 本质 | 数据世界的结构描述 | 业务世界的语义理解 |
| 回答 | 有什么、怎么连、怎么算 | 什么意思、为什么、怎么用 |
| 维护者 | 数据工程师 | 业务分析师 |
| 变更频率 | 低（表结构变了才动） | 高（随时补充业务理解） |
| 删了会怎样 | 查询跑不了 | 查询能跑，但没有业务洞察 |

**核心原则：知识本体不脱离数据本体单独存在。** Ok 的一切价值最终都要通过 Od 落地 — 要么帮助理解 Od 的数据，要么触发 Od 的查询，要么解读 Od 的结果。

### 1.1 Od 的三个概念

| 概念 | 回答 | 对应表 |
|------|------|--------|
| **Property** | 实体有什么（描述性，静态属性） | `ont_object_type`, `ont_property` |
| **LinkType** | 实体之间怎么连（结构性，数据关系） | `ont_link_type` |
| **Metric** | 怎么算（计算性，从属性派生数值） | `ont_metric` |

### 1.2 Ok 的三个概念

| 概念 | 对标 Od | 回答 | 内容 |
|------|---------|------|------|
| **Concept（概念释义）** | Property | 什么意思 | 定义、口径、正常范围、易混淆项 |
| **Causality（因果关系）** | LinkType | 为什么 | 概念之间的因果/相关/组成关系 |
| **Playbook（分析策略）** | Metric | 怎么用 | 多步分析框架，每步可触发数据查询 |

**对称关系：**

```
     Od（数据世界）          Ok（知识世界）
    ┌─────────────┐       ┌─────────────┐
    │  Property    │ ←───→ │  Concept    │
    │  是什么       │       │  什么意思    │
    ├─────────────┤       ├─────────────┤
    │  LinkType    │ ←───→ │  Causality  │
    │  怎么连       │       │  为什么      │
    ├─────────────┤       ├─────────────┤
    │  Metric      │ ←───→ │  Playbook   │
    │  怎么算       │       │  怎么用      │
    └─────────────┘       └─────────────┘
```

### 1.3 三种关系

Od 内部、Ok 内部、Od 与 Ok 之间各有一种关系，互不交叉：

| 关系 | 连接 | 性质 |
|------|------|------|
| **LinkType** | Od ↔ Od | 数据 join 关系（已有） |
| **Causality** | Ok ↔ Ok | 知识间的因果/相关/组成 |
| **Grounding** | Ok → Od | 知识锚定到数据实体 |

**Causality 是纯 Ok 内部关系，不直接触碰 Od。** 例如 "促销力度导致毛利率下降" 连接的是 Ok:Concept("促销力度") 和 Ok:Concept("毛利率")，而不是直接连 Od:ont_metric。Ok 中的每个 Concept 各自通过 Grounding 锚定到对应的 Od 实体。

## 2. LLM 可见性：融合视图

Od 和 Ok 的分离是给人维护的，不是给 LLM 看的。LLM 看到的是融合后的完整对象：

```
对象: 毛利率
  计算: (营收 - 成本) / 营收 × 100%        ← Od:ont_metric
  正常范围: 30%-45%                        ← Ok:Concept
  影响因素: 促销力度(负向), 原材料成本(正向)  ← Ok:Causality
  分析方法: 下降诊断框架(5步)               ← Ok:Playbook
  可用查询: trend, grouped, comparison      ← Od:ont_method
```

**Ok 不膨胀对象数量，它增加每个对象的信息密度。** 假设 50 个 Ok + 20 个 Od，LLM 看到的仍然是 ~20 个对象，只是部分对象变"厚"了。

### 2.1 维护与消费分离

```
维护时（分开）：              消费时（融合）：

Od:ont_metric ─┐              ┌─────────────────┐
               ├─ 融合渲染 ──→│  毛利率          │
Ok:Concept    ─┤              │  计算+含义+因果  │
Ok:Causality  ─┤              │  +可用查询+分析法 │
Ok:Playbook   ─┘              └─────────────────┘
                                      │
                                  注入 LLM context
```

| | Od | Ok | 融合视图 |
|--|----|----|---------|
| 存储 | `ont_*` 表 | `ont_knowledge` + `ont_causality` | — |
| 维护 | 数据工程师 | 业务分析师 | — |
| API | `/api/ontology/objects` | `/api/ontology/knowledge` | 运行时生成 |
| LLM 可见 | 否 | 否 | **是 — 只看这一层** |

## 3. 主题域索引（Topic）

### 3.1 为什么需要 Topic

Ok 和 Od 的数量比例不固定：

- Ok >> Od（知识密集）：3 个数据实体，每个下面 30 条知识 → 以 Od 为骨架会导致单个节点膨胀
- Od >> Ok（数据密集）：50 个数据实体，只有 5 条知识 → 以 Od 为骨架全是空壳
- 两者都很多：Ok 和 Od 各几百条 → 任何一方做骨架都扛不住

**不应该用 Od 或 Ok 任何一方的结构去组织另一方。** 需要一个独立于两者的、稳定的第三层组织结构。

### 3.2 Topic 的设计

Topic 按**业务域**划分，数量由业务本身决定（通常 5-15 个），与 Ok/Od 的数量无关。Ok 和 Od 各自独立挂到 Topic 上。

```
Topic 层（稳定，5-15个）
  │
  ├── 盈利分析
  │     ├── Ok: 毛利率概念、净利率概念、成本因果...
  │     └── Od: 毛利率metric、净利率metric、成本object...
  │
  ├── 客户经营
  │     ├── Ok: 客户分群策略、流失预警框架...
  │     └── Od: 客户object、订单object...
  │
  └── 供应链
        ├── Ok: 库存周转概念...
        └── Od: 库存object、供应商object...
```

### 3.3 不同比例下的自适应

| Ok:Od 比例 | Topic 内表现 | 翻书节奏 |
|-----------|-------------|---------|
| 50:20（知识密集） | 知识列表为主，Od 少量 | 多翻知识再查 |
| 20:50（数据密集） | Od 列表为主，仅有的 Ok 标注 | 少翻直接查 |
| 500:500（都很多） | Topic 内部再分子主题 | 逐层展开 |

**Topic 层自动平衡了知识和数据的比例问题。** 不管下面挂的是知识还是数据、比例如何，LLM 在 L0 看到的永远是 5-15 行的业务域列表。

## 4. Agent 翻书机制

### 4.1 核心思路

传统 RAG 用向量距离做检索决策，本方案将检索决策权交给 LLM 的语义理解。Agent 像翻书一样逐层深入：先看目录，判断该翻哪章，再看细节。

优势：
- 检索是语义推理，不是距离计算 — LLM 理解"下降"意味着需要找对比类知识
- 渐进式深入 — 第一层没找到可以换方向，不像 top-k 一锤子买卖
- LLM 的分类判断能力远强于向量相似度

### 4.2 分层索引

```
L0: Topic 列表             → 固定在 system prompt（5-15行）
L1: Topic 下的 Ok/Od 列表  → lookup(topic) 返回
L2: 具体 Ok 条目或 Od 详情  → read(id) 返回
L3: 数据查询与执行          → query → smartquery
```

### 4.3 工具定义

```
tools:
  lookup(category)       翻目录，返回某个 Topic 下的 Ok/Od 列表
  read(entry_id)         读 Ok 具体知识内容，或 Od 实体详情
  request_query(intent)  表达查询意图（自然语言），系统加载对应的 Od 上下文，返回可用参数
  smartquery(params)     填精确参数，执行 DAX/SQL 查询
```

前两个工具在 Ok 层活动（理解），后两个在 Od 层活动（执行）。`request_query` 是 Ok 到 Od 的桥梁。

### 4.4 Od 从 prompt 层下沉到 tool 层

**关键架构转变：Od 不再注入 LLM 的 system prompt，而是成为 Tool 的内部实现。**

```
之前（Od 在 prompt 层）：
  System Prompt: [全量 Od 对象/属性/指标]
  LLM: 看到词汇表 → 匹配 → 填参数
  → LLM 在做词汇匹配，不是在理解业务

之后（Ok 在 prompt 层，Od 在 tool 层）：
  System Prompt: [Ok 知识目录]
  LLM: 理解业务 → 翻知识 → 形成意图 → 调工具
  Tool 内部: 拿到意图 → 查 Od 解析参数 → 执行
  → LLM 用 Ok 思考，Tool 用 Od 执行
```

## 5. 完整对话流程

### 5.1 理论 Turn 顺序

```
Turn 1 — Ok 定位（翻目录）
  输入: [system] [tools] [Topic-L0目录]
  用户: "毛利率是不是有问题？"
  LLM: 需要了解毛利率 → lookup("盈利分析")

Turn 2 — Ok 理解（读知识）
  输入: [system] [tools] [Topic-L0] [L1:盈利分析下的条目列表]
  LLM: 找到毛利率相关知识 → read("毛利率概念") + read("毛利率因果")
  获得: 正常范围30-45%，受促销影响，有下降诊断框架

Turn 3 — Od 加载（桥接到数据）
  输入: [system] [tools] [Ok上下文] [Od:毛利率metric + 可用维度 + 方法]
  LLM: 知道要看趋势 → request_query("毛利率近6个月趋势")
  系统: 返回 metric=毛利率, method=trend 可用, 维度有月份/区域/促销类型

Turn 4 — Od 执行（查询数据）
  LLM: 选定参数 → smartquery({metric: "毛利率", method: "trend", groupBy: "月份"})
  系统: 拼 DAX/SQL → 执行 → 返回结果

Turn 5 — Ok 解读（结合知识解释结果）
  LLM: 28%，连续3月下降，从35%→28%
       结合 Ok 知识：低于正常范围(30-45%)
       因果关系提示检查促销力度
       → 给出业务解读 + 建议下一步分析
```

### 5.2 对比现有流程

```
现在（1 turn，纯 Od）：
  [system + 全量Od词汇表] → LLM 词汇匹配 → smartquery(参数) → 结果
  缺陷：Od 全量注入占 context，LLM 只做匹配没有理解

之后（多 turn，Ok 驱动）：
  [Ok目录] → [Ok知识] → [精准Od] → smartquery → [Ok解读]
  每一步缩小范围，每一步增加理解
```

### 5.3 自适应行为

LLM 不一定每次都走完全部 Turn，它会根据情况跳步：

| 场景 | 实际路径 | 原因 |
|------|---------|------|
| 简单查询 "销售额多少" | Turn1 → Turn3 → Turn4 | 不需要深入知识 |
| 解释性问题 "毛利率什么意思" | Turn1 → Turn2 → 直接回答 | 不需要查数据 |
| 诊断性问题 "毛利率为什么下降" | Turn1-5 全部走完 | 需要知识+数据+解读 |
| 重复追问 "再看看按区域" | Turn3 → Turn4 | Ok 上下文已在，直接查 |

## 6. 知识构建路径

### 6.1 从现有数据本体生长

不从零开始，而是从 Od 自动生成 Ok 骨架，人工补充血肉：

**第一步：从 ont_metric 生成 Concept 骨架**
- 每个 metric 自动生成一个 Concept 条目
- 预填：名称、计算口径（从 dax_expression）
- 人工补充：业务含义、正常范围、易混淆项

**第二步：从 ont_query_log 挖掘 Causality**
- 高频查询组合暗示业务关注点
- 失败查询暗示知识缺口
- LLM 辅助提取因果关系候选，人工审核

**第三步：从领域专家沉淀 Playbook**
- 访谈业务分析师的分析思路
- LLM 辅助结构化为步骤框架
- 每步关联到具体的 Od 查询

### 6.2 持续积累

知识本体通过数据飞轮持续增长：

```
用户提问 → Agent 回答 → 用户反馈
                            │
                            ▼
                  ont_query_feedback
                            │
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
      发现新概念       发现新因果       发现新分析模式
      → 补充Concept   → 补充Causality  → 补充Playbook
```

## 7. 设计约束和注意事项

1. **L0 目录必须控制在 15 行以内** — Topic 数量由业务域决定，不应无限膨胀
2. **每条 Ok 必须有 grounding 锚点** — 全局性知识挂在 version 级别，但不允许完全无锚
3. **Ok 的 summary 是给 LLM 看的** — 要写清楚"这条知识能帮你解决什么问题"，不是给人看的摘要
4. **Agent 最大翻书深度 3-4 层** — 超过说明 Topic 划分或知识组织有问题
5. **删除 Od 实体时审查关联 Ok** — 通过 grounding 关系找到所有引用该实体的知识条目
6. **Causality 只连 Ok，不跨层连 Od** — 保持关系类型的纯粹性
7. **Topic 可以有子 Topic** — 应对单个域下 Ok/Od 过多的情况，但层级不超过 2
